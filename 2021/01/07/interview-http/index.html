HTTP 发展历程

2021-01-07 interview

HTTP 是一个用于传输超文本文档的应用层协议，全名为超文本传输协议

HTTP发展历史如下

HTTP 0.9

HTTP 0.9 也被称为单行协议，最初版本的 HTTP 协议并没有版本号，后来它的版本号被定位在 0.9 ，以区分后来的版本。
HTTP 0.9 极其简单，请求由单行指令构成，以唯一可用方法 GET 开头，其后跟目标资源的路径。
HTTP 0.9 响应内容并不包含 HTTP 头，这意味着只有 HTML 文件可以传送，无法传输其他类型的文件，也没有状态码，出现异常时一个特殊的包含问题描述信息的HTML文件将被响应返回。

特点

  • 只有一个请求行，并没有 HTTP 请求头和请求体。
  • 服务器没有返回头信息，只返回数据信息。
  • 返回的文件内容是以 ASCII 字符流来传输的，因为都是 HTML 格式的文件，所以使用 ASCII 字节码来传输是最合适的。

HTTP 1.0

HTTP 1.0 构建了可扩展性，解决了传输多种类型文件的需求
HTTP 1.0 引入了请求头和响应头，它们都是以为 Key-Value 形式保存的，在 HTTP 发送请求时，会带上请求头信息，服务器返回数据时，会先返回响应头信息。

特点

  • 引入版本号，版本信息会随着每个请求发送，即 HTTP 1.0 被追加到了 GET 行。
  • 引入请求头，在发起请求时候会通过 HTTP 请求头告诉服务器它期待服务器返回什么类型的文件、采取什么形式的压缩、提供什么语言的文件以及文件的具体编码。
  • 引入响应头，服务器以请求头中信息准备数据，并以响应头的信息告诉客户端数据采用何种格式返回。
  • 引入状态码，状态码会在响应开始时发送，使浏览器能了解请求执行成功或失败，并相应调整行为。
  • 引入了缓存机制，通过状态码与 If-Modified-Since 、 Expires 等控制更新或使用本地缓存。
  • 引入了 Content-Type 头，使 HTTP 具备了传输除纯文本 HTML 文件以外其他类型文档的能力。

HTTP 1.1

HTTP 1.1 是标准化的协议，HTTP 1.1 消除了大量歧义内容并引入了多项改进。
默认长连接、支持数据分块传输、断点续传功能

特点

  • 缓存处理，HTTP 1.1 引入了更多的缓存控制策略，例如Entity tag、If-Unmodified-Since、If-Match、If-None-Match等更多可供选择的缓存头来控制缓存策略。
  • 在请求头中引入了 range，它允许只请求资源的某一个部分，便于制作断点续传功能
  • 错误通知的管理，在 HTTP 1.1 中新增了 24 个错误状态码。
  • 增加 Host 请求头，能够使不同域名配置在同一个 IP 地址的服务器上。
  • 支持默认长连接，HTTP 1.1 支持长连接，在同一个连接上可以传输多个请求和响应，一般浏览器对于同一个域名允许同时建立6个长链接。
  • 增加管线化技术，允许在第一个应答被完全发送之前就发送第二个请求，以改善队头阻塞问题，但响应的顺序还是会按照请求的顺序返回。
  • 支持响应分块，通过设置 Transfer-Encoding: chunked 进行分块响应，允许响应的数据可以分成多个部分，配合服务端尽早释放缓冲可以获得更快的响应速度。

HTTP 2.0

网页愈渐变得的复杂，甚至演变成了独有的应用，媒体的播放量，增进交互的脚本大小也增加了许多，更多的数据通过 HTTP 请求被传输，由此 HTTP 2.0 为网络效率做了大量的优化。
主要解决传输问题，将数据量变得更少

特点

  • 二进制分帧，HTTP 2.0 是二进制协议而不是文本协议，将所有传输的信息分割为更小的消息和帧,并对它们采用二进制格式的编码。
  • 多路复用，并行的请求能在同一个链接中处理，在同一域名下所有访问都是从同一个TCP连接中走，HTTP消息被分解为独立的帧，服务端根据标识符和首部将消息重新组装起来，移除了 HTTP 1.1 中顺序和阻塞的约束。
  • **压缩 header**，header 在一系列请求中常常是相似的，其移除了重复和传输重复数据的成本。
  • 服务端推送，服务器可以主动地向客户端推送资源，而无需客户端明确的请求。

HTTP 存在的安全问题

由于 HTTP 本身是明文传输的，没有经过任何安全处理。因此会存在流量劫持的问题

为了解决这个问题，人们提出了 HTTPS
HTTPS 一般理解为 HTTP + SSL/TLS，通过 SSL 证书来验证服务器的身份，并为浏览器和服务器之间的通信进行加密。

一般通信过程如下：

简要描述以上过程
TCP 三次握手

  1. 第一次握手，设置标志位为 SYN，序列号与确认号均为 0
  2. 第二次握手，设置标志位为 SYN、ACK，序列号为 0，确认号为 1
  3. 第三次握手，设置标志位为 ACK，序列号为 1，确认号为 1
    注意：对于三次握手，需要注意的是对于单 ACK 标志，是不会消耗序列号的，因此下次包可以再次复用序列号

TLS 握手

  1. Client Hello (Client -> Server)，此阶段主要向服务端提供参数：客户端版本,客户端随机数,压缩方法列表,对称加密算法列表,扩展列表
  2. Server Hello (Server -> Client)，此阶段主要向客户端提供选择的参数：服务端版本,服务端随机数,对称加密算法,压缩方法,扩展（后三项均从客户端提供的列表进行挑选）
  3. Server Certificate (Server -> Client)，此阶段服务端向客户端提供证书
  4. Client Certificate (Client -> Server)，此阶段可选，客户端向服务端提供证书
  5. Server Key Exchange (Server -> Client)，此阶段只有在客户端无法依据第三步的证书进行生成必须密钥时才会进行，使用的临时的 DH/ECDH 密钥协商算法，证书中是不包含这些动态的 DH 信息(DH 参数和 DH 公钥)，所以需要使用 Server Key Exchange 消息传递这些信息。传递的动态 DH 信息需要使用 Server 私钥进行签名加密。
  6. Server Hello Done (Server -> Client)，服务端向客户端确认完成
  7. Client Key Exchange (Client -> Server)，此阶段 client 将验证证书，使用之前选择的对称加密算法生成 pre-master-secret.使用服务端的公钥进行加密，随后发送给服务端。（此内容只有服务端才能解密，因此双方现在均拥有，两个随机数+pre-master-secret）（当 Client 使用一个动态的 Diffie-Hellman 指数时，这个消息就会包含 Client 的 Diffie-Hellman 公钥。如果 Client 正在发送一个包含一个静态 DH 指数(例如，它正在进行 fixed_dh Client 认证)的证书时，这个消息必须被发送但必须为空。）
  8. Client Change Cipher Spec (Client -> Server)，客户端声明准备使用协商好的公共密钥进行加密（公共密钥 = 两个随机数+pre-master-secret)
  9. Client Handshake Finished (Client -> Server)，客户端握手阶段完成
  10. Server Change Cipher Spec (Server -> Client)，服务端声明准备使用协商好的公共密钥进行家吗（公共密钥 = 两个随机数+pre-master-secret)
  11. Server Handshake Finished (Server -> Client)，服务端握手阶段完成

TCP 四次挥手

  1. 第一次挥手，设置标志位为 ACK,FIN，进入 FIN_WAIT1 状态
  2. 第二次挥手，设置标志位为 ACK，进入 CLOSE_WAIT 状态
  3. 第三次挥手，设置标志位为 FIN，进入 LAST_WAIT 状态（接收到 ACK 为 CLOSED 状态
  4. 第四次挥手，设置标志位为 ACK，进入 TIMEWAIT 状态（2个 RTT 后进入 CLOSED 状态）
